# 文档概述（订阅后自动发券）

- 基于现有 Redis/Lua 秒杀预扣与 Kafka 异步落单机制，采用“订阅 + 自动发券”策略：当库存为 0 时，用户订阅到券提醒；一旦库存回流或补仓，系统按订阅队列顺序自动为用户创建订单，创建成功后再通知用户，无需用户主动点击领取。
- 保持公平先来先得、一人一单与幂等一致性；在自动发券失败时进行回滚并继续匹配下一位订阅用户。

# 业务目标

- 库存为 0 时，允许未购买用户订阅到券提醒（可设订阅有效期）。
- 当库存回流/补仓时，系统按订阅队列自动发券（创建订单），随后向命中用户推送到券成功通知。
- 保证一人一单、幂等与一致性；主链路仍由 Kafka 异步落单与 DB 扣减完成。

# 前端交互（ShopDetail.vue）

- 初始化一次性查询每张券的状态：
  - 已购：按钮显示“已领取”。
  - 库存 > 0：显示“限时抢购”。
  - 库存 <= 0 且未购：
    - 未订阅：显示“到券提醒”按钮。
    - 已订阅：置灰为“已订阅提醒”，可提供“取消订阅”。
- 自动发券：当库存回流/补仓并命中订阅队列时，后端自动创建订单并通知用户；前端无需用户点击领取，收到状态后展示“已领取”。
- 建议使用批量接口获取 `isPurchased + subscribeStatus + stock`，避免 N+1。

# Redis 与 Lua 设计（订阅队列 + 自动匹配）

- 键约定（每个 `voucherId` 一套，保持 Hash Tag 同槽位）：
  - `SECKILL_SUBSCRIBE_USER_TAG_KEY = "seckill:subscribe:user:{%s}"`（SET：订阅用户集合，去重）
  - `SECKILL_SUBSCRIBE_ZSET_TAG_KEY = "seckill:subscribe:z:{%s}"`（ZSET：订阅队列，成员 `userId`，score 为入队时间，保证公平）
  - `SECKILL_SUBSCRIBE_STATUS_TAG_KEY = "seckill:subscribe:status:{%s}"`（HASH：`userId -> SUBSCRIBED | SUCCESS | UNSUBSCRIBED`）
  - 复用：`SECKILL_STOCK_TAG_KEY`（库存）、`SECKILL_USER_TAG_KEY`（一人一单标记）
- TTL 设计：键 TTL = `max(券结束时间 - now, 1s)`；状态键也随活动结束自动过期。
- Lua 操作建议：
  - `subscribe_join.lua`：校验活动时间与已购；`SADD` + `ZADD` 入队；`HSET status SUBSCRIBED`。
  - `subscribe_cancel.lua`：`SREM` + `ZREM` 出队；`HSET status UNSUBSCRIBED`。
  - `subscribe_match.lua`（在库存回流/补仓触发）：
    - 循环：尝试 `DECR stock`，若成功则 `ZPOPMIN` 取队首；校验队首用户未购（`SECKILL_USER_TAG_KEY`）；标记一人一单；输出 `userId + voucherId + traceId` 给业务层进行异步落单；若队列为空则 `INCR stock` 回滚。
    - 返回已匹配的用户列表（供消息生产者发送 Kafka）。

# 自动发券与通知

- 触发时机：
  - 运营补仓（`updateSeckillVoucher` 修改 `stock`）
  - 订单取消/退款回库（`VoucherOrderServiceImpl#cancel`）
- 执行流程：
  - 发布“库存回流/补仓”事件到主题（建议 `SECKILL_VOUCHER_RESTOCK_TOPIC`）：`voucherId, stockDelta, ts`
  - 消费端执行 `subscribe_match.lua`，批量取出命中的订阅用户，逐个发送 `SeckillVoucherMessage` 到原下单主题进行异步落单（复用现有 `SeckillVoucherProducer`/`Consumer`）。
  - 订单创建成功后，发送“到券成功”通知到 `SECKILL_VOUCHER_NOTIFY_TOPIC`（或统一通知服务），渠道可为站内信/短信/公众号。
- 通知准则：先自动发券成功，再通知用户；失败会回滚 Redis 并继续匹配下一位订阅用户。

# 幂等与一致性

- 订阅与取消：`SADD/SREM` 幂等；状态 HASH 与 ZSET 保持同步。
- 自动发券：通过 `SECKILL_USER_TAG_KEY` 防止重复发券；Kafka 消费端以 `voucherId:userId` 幂等键控制重复消息。
- Lua 原子匹配：扣减与队列弹出在同脚本内执行，避免竞态；落单失败由消息侧回滚 Redis 并继续匹配。
- 通知：以 `voucherId:userId:orderId` 做 `SETNX` 去重与 TTL，避免重复通知。

# 风控与边界

- 订阅限频：`userId` 维度每日订阅次数限制；同券仅可订阅一次。
- 自动发券公平性：按入队时间戳排序（ZSET）；如引入 VIP 权重，可调整 score，需明确规则与审计。
- 渠道限流：批量通知时做速率限制与失败重试（指数退避）。
- 异常场景：自动发券落单失败/库存不足/用户已购变化，均走回滚与下一位匹配逻辑；取消订阅与匹配并发时以 Lua 脚本原子处理。

# 监控与指标

- 指标建议：
  - 订阅总量、有效订阅量、通知到达率、点击率、领取成功率
  - 回库来源占比（取消/补仓）
- 告警建议：
  - 通知失败率异常、订阅键过期异常、库存回流后通知缺失

# 与“候补排队自动发券”方案的差异

- 不自动创建订单：避免用户意愿不明确带来的争议与风控复杂度；适配“免费券直接领取”的典型场景。
- 成本更低：无需维护候补队列的公平策略、并发匹配与幂等落单；仅做订阅集合与通知。
- 仍可拓展：如需“保留权”，可在提醒中提供短时保留权（`SETNX` 标记 + TTL），但当前版本不默认实现。
