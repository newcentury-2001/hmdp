**目标**
- 保证“(user_id, voucher_id)”唯一约束在分库分表后仍有效，同时兼顾高并发写入的吞吐与热点分散。
- 让同一对 `(user_id, voucher_id)` 的写入始终落到同一个物理库表，从而用本地唯一索引即可拦截重复。

**推荐分片方案**
- 库按 `voucher_id` 分片，表按 `user_id` 分片
  - 数据库分片键：`voucher_id`，保证同一券的库存记录与订单在同库，便于“先扣库存再插单”的事务与关联。
  - 表分片键：`user_id`，将同库内的订单写入分散到多张分表，降低单表热点。
  - 唯一性保障：同一对 `(user_id, voucher_id)` 固定路由到同一个“库 + 表”，在该物理表上建立本地唯一索引 `uk_user_voucher (user_id, voucher_id)` 即可。
- 若担心单券极端热点集中到一个库，可在“表分片”使用复杂分片算法（同时用 `user_id` 和 `voucher_id` 参与）在该库内均衡到多张分表：
  - 表分片键：复杂分片，键集合为 `{user_id, voucher_id}`，计算 `index = f(user_id, voucher_id) % tableCount`。
  - 唯一性保障：对同一 `(user_id, voucher_id)`，算法返回固定表号，因此本地唯一索引仍有效。

**唯一索引与路由原则**
- 分库分表后无法做“跨库跨表的全局唯一索引”，所以必须通过分片算法保证“同一对 `(user_id, voucher_id)` 一定路由到同一物理表”，以便用本地唯一索引拦截重复。
- 选择分片键的判断：
  - 以 `voucher_id` 做库分片：库存更新与订单写入能在同库内完成事务；Kafka 分区也可用 `voucher_id` 对齐，避免跨库竞争。
  - 以 `user_id` 做表分片或复杂分片：同库内扩展并发、分散行级锁。
- SQL 必须携带分片键
  - 写入和查询都应包含 `voucher_id`（库分片）与 `user_id`（表分片/复杂分片）条件，否则 ShardingSphere 可能广播到所有库表，影响性能与一致性。

**ShardingSphere 配置示例（库按 voucher_id、表按 user_id）**
- 逻辑表：`voucher_order`
- 物理库：`ds0 ... ds3`（示例 4 库）
- 物理分表：`voucher_order_0 ... voucher_order_15`（每库 16 表）
- 数据库分片策略
  - `standard`，`shardingColumn: voucher_id`，`INLINE` 算法：`ds$->{voucher_id % 4}`
- 表分片策略
  - `standard`，`shardingColumn: user_id`，`INLINE` 算法：`voucher_order_$->{user_id % 16}`
- 唯一索引
  - 在每个物理分表上建立：`uk_user_voucher(user_id, voucher_id)`

示意：
- 数据库路由：`ds${voucher_id % 4}`
- 表路由（同库内）：`voucher_order_${user_id % 16}`
- 同一对 `(user_id, voucher_id)` 的重复写入，必然命中同一物理表，唯一索引生效。

**复杂分片算法示例（同库内用复合键分表）**
- 表分片算法（Java）使用 `ComplexKeysShardingAlgorithm`：
  - 取 `u = user_id`，`v = voucher_id`
  - 计算 `index = ((v * 1315423911) ^ u) % tableCount`
  - 返回 `voucher_order_${index}`
- 配置使用 `complex` 策略，`shardingColumns: user_id, voucher_id`，算法名指向上述实现。
- 作用：对同券的海量写入在该库内均匀打散到多张分表，降低单表锁竞争，同时保证同一 `(user_id, voucher_id)` 始终落到同一张表。

**注意事项**
- 事务边界
  - 库分片用 `voucher_id` 能让“扣库存 + 插单”同库事务完成；避免跨库分布式事务。
- 唯一索引建在物理表
  - 在所有物理分表执行 DDL 建立 `uk_user_voucher`。
- Kafka 分区与 DB 分片对齐
  - 建议 Kafka `key` 使用 `voucher_id`，使同券消息落到同一消费者实例，配合同库分片减少锁争用。
- 读写路由
  - 所有写 SQL 必须包含 `voucher_id` 与 `user_id`；查询也尽量携带这两个条件，避免广播扫描。
- 迁移与扩容
  - 分库扩容时保持算法稳定（如取模数改变需做数据迁移）；可用一致性哈希或预留分片位减小迁移成本。
